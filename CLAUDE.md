# Choonsikmom Archive — 게시글 작성 규칙

이 저장소의 게시글을 통일된 형식으로 작성하기 위한 규칙이다. 새 글을 쓰거나 기존 글을 고칠 때 이 규칙을 따른다.

## 1. 문체

- **서술 문장은 `-다` 평서체**로 간결하게 끝낸다. 정중체(`~합니다`, `~입니다`)를 쓰지 않는다.
  - `표현됩니다` → `표현된다`
  - `필요합니다` → `필요하다`
  - `~라고 할 수 있습니다` → `~라고 할 수 있다`
- **번호·글머리(리스트) 항목은 명사·구로 끝내도 된다.** 문장으로 끝나지 않고 단어/구로 끝나는 현재 형식을 유지한다.
  - 예: `- 위치(Translation) 3자유도`, `- 서브픽셀 정확도의 코너 검출`
- 군더더기 없이 핵심만. 한 문단은 짧게.
- **주요 기술 용어는 영어 원어로 쓴다.** 한글 음차(`마칭 큐브`, `포인트 클라우드`, `복셀`, `래스터화` …)나 억지 번역어로 풀지 않는다.
  - 예: `3D 재구성` → `3D Reconstruction`, `마칭 큐브` → `Marching Cubes`, `포인트 클라우드` → `point cloud`, `복셀` → `voxel`, `아웃라이어` → `outlier`, `다운샘플링` → `downsampling`, `래스터화` → `rasterization`
  - 처음 나올 때 필요하면 짧은 한글 설명을 괄호로 덧붙일 수 있다: `voxel(공간을 나눈 3D 격자 단위)`. 반대로 `복셀(voxel)`처럼 **음차+영어를 겹쳐 쓰지 않는다.**
  - 서술어·조사는 그대로 한글 `-다`체를 쓰고, 영어는 **용어(명사)에만** 적용한다. 이미 자연스럽게 굳은 일반어(`렌더링`, `노이즈` 등)는 예외로 둔다.
  - 반대로 `법선(normal)`, `광선(ray)`처럼 **일반 한글어 + 영어 용어** 병기는 유지해도 된다.

## 2. Front matter

```yaml
---
title: "[공부] 제목"          # 또는 [논문리뷰]
excerpt: "한 줄 요약"
categories:
  - 공부                      # 또는 논문리뷰 (메인 카테고리)
  - CV                        # 세부 분야: AI / CV / NLP / RL / Robotics
tags:
  - 키워드1
  - 키워드2
toc: true
toc_sticky: true
---
```

- **파일명**: `YYYY-MM-DD-slug.md` (slug은 영문 소문자-하이픈).
- **글은 미래 시각이면 숨겨진다**(`future: false`). 그래서 시리즈 순서를 날짜로 조절할 때 **오늘보다 미래 날짜를 쓰지 않는다.**
  - **여러 날에 걸친 시리즈**: 시각 없이 날짜(일 단위)만 오름차순으로 둔다(1편이 가장 과거). 예: 6DoF 시리즈.
  - **같은 날 한 번에 올리는 시리즈**: 같은 날짜에 몰면 목록 순서가 뒤섞이므로, `date:`에 **오름차순 시각**을 준다(1편이 가장 이른 시각). 단 **반드시 타임존 오프셋 `+0900` 을 붙인다** — 오프셋 없는 `2026-07-05 12:00:00` 은 Jekyll이 UTC로 해석해, 한국 기준 과거여도 미래로 판단해 숨길 수 있다. 모든 시각은 **현재 시각보다 과거**여야 한다(`date` 명령으로 확인).
- 첫 카테고리가 메뉴 분류(공부/논문리뷰), 둘째가 세부 분야(사이드바 서브카테고리)를 결정한다.

## 3. 글 구조

1. 인용문(`>`) 한두 줄로 리드 — 이 글이 무엇인지/시리즈 위치
2. 번호 붙인 섹션(`## 1.`, `## 2.` …)
3. 필요하면 비교 **표**로 정리
4. 마지막에 `## 정리` + 다음 글 링크

## 4. 내부 링크 (시리즈)

- 다른 글로 링크할 때 `{% raw %}{% post_url YYYY-MM-DD-slug %}{% endraw %}` 를 쓴다.
- 시리즈 글은 **끝에 다음 글 링크**를 단다. 날짜는 읽는 순서대로 오름차순(1편이 가장 과거)으로 둬서 시간 순서와 읽는 순서를 맞춘다.

## 5. 수식

- **인라인·블록 모두 `$$ ... $$` 로 감싼다.** MathJax로 렌더한다.
  - 인라인: 텍스트와 같은 줄에 둔다. 예: `... 회전행렬 $$R \in SO(3)$$ 로 ...`
  - 블록: `$$` 를 앞뒤 **독립된 줄**에 두고 그 사이에 수식을 쓴다.
- **인라인에 홑 `$ ... $` 를 쓰지 않는다.** kramdown(GFM)은 홑 달러 사이를 수식으로 인식하지 않아, 수식 안의 `_`(밑줄)를 강조(`<em>`)로, `|`(파이프)를 표 구분자로 잘못 해석해 **수식이 깨진다**. `$$ ... $$` 로 감싸야 kramdown이 `\( \)`/`\[ \]` 로 변환해 내용을 보호한다.
- **조건부·구분 기호는 홑 파이프 `|` 대신 `\mid` 를 쓴다.** `\pi(a_t|s_t)` → `\pi(a_t \mid s_t)`. 홑 `|` 는 그 줄 전체를 표로 만들어 버린다(노름 `\| ... \|` 는 이미 안전).
- **수식은 던지지 말고 자세히 설명한다.** 수식을 쓰면 **등장하는 모든 기호가 무엇인지** 말로 풀어 준다. 무엇을 최소화/최대화하는지, 각 항이 무슨 역할인지, 왜 그런 형태인지 한두 줄로 잇는다.
  - **처음 나오는 변수·기호는 반드시 정의한다.** 문자·아래첨자·집합·연산자 모두. 예: `$$H = \sum_i x_i y_i^\top$$ 에서 $$x_i$$ 는 중심 정렬된 측정점, $$y_i$$ 는 기준점, $$H$$ 는 둘의 교차공분산 행렬($$3\times3$$)이다.`
  - 차원·정의역도 밝힌다: `$$R \in SO(3)$$ (3×3 회전행렬)`, `$$i = 1, \dots, N$$ (점 쌍의 인덱스)`.
  - "이 식은 A를 최소화한다", "앞 두 항은 상수라 무시된다", "등호는 ~일 때 성립한다"처럼 **식의 의미와 흐름**을 붙인다. 유도는 한 단계씩 근거를 남긴다.
  - 기호를 정의 없이 갑자기 쓰지 않는다. 읽는 사람이 식만 보고 "이 $$k$$ 가 뭐지?" 하지 않도록 한다.

## 6. 그림 (SVG 다이어그램)

개념 이해를 돕는 **SVG 다이어그램**을 적극적으로 넣는다. `assets/images/`에 `.svg`로 저장하고 마크다운으로 삽입한다.

```markdown
![설명 텍스트](/assets/images/diagram-name.svg)
```

### SVG 스타일 가이드 (통일성)

- 밝은 배경 기준, 깔끔한 sans-serif. `font-family="Pretendard, 'Malgun Gothic', sans-serif"`
- **헤더 pill**(둥근 사각형)로 항목을 구분하고, 파스텔 색으로 구별한다.
  - 파랑 계열: 배경 `#e9f1fb` / 테두리 `#b9d0ec` / 글자 `#2c5aa0`
  - 주황 계열: 배경 `#fbeed6` / 테두리 `#e6cfa0` / 글자 `#b5761a`
  - 초록 계열: 배경 `#e6f4ea` / 테두리 `#b7dcc0` / 글자 `#2e7d46`
- 내용 박스는 **점선 둥근 사각형**(`stroke:#c9ccd1; stroke-dasharray:5 4; fill:none`).
- 흐름은 **회색 화살표**(`stroke:#8a9199`)로 연결한다.
- 본문 글자색 `#333`, 보조 설명은 회색 `#6b7178`.
- `viewBox`를 지정해 반응형으로 스케일되게 한다.

> 참고 기준: TSDF vs 3DGS 비교처럼, 두 개념을 좌우로 나눠 헤더 pill + 점선 박스 + 화살표로 구성하는 형식.

## 7. 논문리뷰 글 규칙

메인 카테고리가 `논문리뷰` 인 글은 위 규칙(1~6)에 더해 다음을 지킨다.

### 7.1 논문 정보 표

리드(인용문) 다음, 첫 번호 섹션(`## 1.`) 앞에 `## 논문 정보` 표를 둔다. **이 구조를 유지한다.**

```markdown
## 논문 정보

| 항목 | 내용 |
| --- | --- |
| 제목 | [논문 제목](논문 URL) |
| 저자 | 저자 나열 |
| 소속 / 연도 | 소속, 연도 (arXiv:번호) |
| 분야 | 세부 분야 |
| 코드 | [GitHub](GitHub URL) |
```

- **제목 행은 논문 URL(arXiv 등 아카이브)로 링크**한다.
- **공식 코드 저장소가 있으면 `코드` 행을 추가**해 GitHub URL로 링크한다. 없으면 이 행을 뺀다.
- 위 4개 행(제목·저자·소속/연도·분야)은 기본으로 유지하고, 필요하면 항목만 덧붙인다.

### 7.2 원문 섹션 구조를 유지한다

- **리뷰의 섹션 구성은 원 논문 본문의 섹션 구조를 그대로 따른다.** 논문에 있는 섹션을 빠뜨리지 않고, 논문에 없는 섹션을 새로 만들지 않는다.
  - 논문이 **Conclusion(결론)** 으로 끝나면 리뷰도 결론 섹션으로 끝낸다. 원문에 없는 마무리 섹션(예: "총평", "Takeaway")으로 대체하지 않는다.
  - 예: PPO 논문 = 1 배경 → 2 Background → 3 Clipped Objective → 4 Adaptive KL → 5 Algorithm → 6 Experiments → 7 Conclusion. 리뷰도 이 순서·번호를 따른다.

### 7.3 논문 본문에 없는 비평을 넣지 않는다

- **원 논문에 없는 평가·의견(강점, 약점, 총평, "~가 아쉽다", "영향력이 크다" 등)을 리뷰에 넣지 않는다.** 「강점」「약점」 같은 섹션을 만들지 않는다.
- 본문 서술은 **논문이 실제로 말한 내용의 요약·설명**에 한정한다. 저자의 주장·실험 결과·설계 의도를 전달하되, 리뷰어의 판단을 덧붙이지 않는다.

## 8. 인터랙티브 자료 (공부 글 한정)

- **공부(카테고리 `공부`) 글에 한해**, 정적 그림(SVG)만으로 이해가 어려운 개념에 인터랙티브 자료(애니메이션·토글·슬라이더 등)를 넣는다. 논문리뷰 글에는 넣지 않는다.
- **꼭 필요한 곳에만 넣는다. 남발하지 않는다.** "움직임·상호작용이 있어야 이해되는" 개념(렌더링·샘플링 과정, 알고리즘 단계, 파라미터에 따른 결과 변화 등)에만 쓰고, 표·SVG로 충분한 곳엔 쓰지 않는다. 한 글에 하나면 충분한 경우가 많다.
- **구현 방식**: 위젯을 `_includes/`에 HTML 파일로 두고, 글에서 `{% raw %}{% include 위젯이름.html %}{% endraw %}` 로 불러온다(마크다운 파서가 코드를 건드리지 않게).
  - **자기완결적(self-contained)**: 인라인 `<style>` + `<script>` + `<canvas>`(또는 SVG)로 만들고 외부 라이브러리·네트워크 의존성을 두지 않는다.
  - **Liquid 충돌 방지**: include 파일 안에 `{{` 또는 `{%` 문자열을 쓰지 않는다.
  - **컨트롤 제공**: 재생/일시정지·리셋 같은 조작과 현재 상태 표시를 넣는다. 위젯은 둥근 모서리 카드로 감싸 본문과 구분한다.
